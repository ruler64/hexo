# 项目总结与 ES 模块工作计划

# 一、项目整体架构概览

项目名称：hmall 高并发电商系统（Spring Cloud Alibaba 微服务）

核心服务：

  • trade-service ：下单、关单、对账

  • item-service ：商品库存、缓存预热、Canal 监听

  • pay-service ：支付

  • cart-service ：购物车

# 二、已完成的核心模块

# 下单链路（高并发核心）

技术栈：RocketMQ 事务消息 + Redis Lua 原子扣减 + Canal 推送消息给下游服务 + ShardingSphere 主从分离

流程：

用户下单

  → 构建 LuaExecutionContext（KEYS 结构见下）

  → luaRocketMQTemplate.sendMessageInTransaction()

  → 半消息发送成功后回调 OrderTransactionListener.executeLocalTransaction()

  → 执行 batch_deduct_stock.lua（原子扣减库存 + 写幂等key + 写反查flag key + 更新per-item版本号）

  → Lua 成功 → COMMIT → OrderRocketMQConsumer 消费消息 → 写 MySQL order(id幂等) + order_detail + local_event_outbox + 尝试在afterCommit中直接发送消息到rocketMQ (outbox用于通知下游服务的兜底表，canal监听并广播推送给"延迟关单"、"商品扣减库存"、"购物车清空"服务+xxlJob兜底处理)

通知下游服务的功能原本由 xxlJob 进行，但是考虑到实时性，和 xxlJob 高频扫表带来的数据库负载，使用 canal 捕捉库存变化来推送消息给下游服务，符合 CDC 标准的开发方式，加快补偿路径的补偿速度；将 xxlJob 转向 canal 不可用时的兜底服务，降低扫表带来的数据库压力。

Tips：

afterCommit 里的数据库更新操作隐藏着 “死锁” 和 “连接池枯竭” 风险 你在 afterCommit （事务提交后）里调用了 localEventOutboxMapper.updateStatus(outboxId, 1); 。

  • 隐患： afterCommit 执行时，原先的数据库连接可能已经被 Spring 释放或者处于特殊状态。在此处再次执行数据库写操作，它会重新发起一次独立的 Auto-Commit 事务。在高并发下，频繁在事务外钩子里触发新的 DB 连接，容易导致 Druid 连接池瞬间被挤爆。
  • 标准做法： afterCommit 里面只做纯网络 IO 事情（比如发 RocketMQ）。至于更新本地消息表状态，通常不放在核心线程里同步做。
    • 更优解：MQ 发送成功后，将更新本地消息表交给线程池异步去做，进一步加快主线程的执行速度。本地消息表的初始状态就是 0（未发送） 。XXL-Job 每隔 10 秒扫一次 1 分钟前创建且状态为 0 的数据进行补偿。如果 MQ 已经成功了，下游消费者天然做好了幂等性，哪怕 XXL-Job 重发一次也无所谓（At-least-once 语义）。
  • 异步落库中的 itemClient.queryItemByIds 属于分布式巨坑（RPC 事务内网络震荡） handleDbOrder 方法加上了 @Transactional 事务注解。但是在事务内部，你调用了 itemClient.queryItemByIds(itemIds) 这个 Feign 远程调用（RPC）。
  • 致命隐患：绝对不要在本地事务内部包含 RPC 远程调用或者慢网络 IO！
    • 每一个 @Transactional 都会霸占一个 MySQL 的数据库连接。如果下游的 item-service 此时发生毛刺、网络拥堵，RPC 耗时从 5ms 飙升到 3 秒，那么这个本地事务就会卡住 3 秒。
    • 带来的后果是：MySQL 连接被严重长时间占用，高并发下单时，几百个线程瞬间把数据库连接池卡死，整个微服务直接瘫痪（连接池等待超时）。
  • 正确重构方案：把 RPC 调用移到事务外面。

# 库存 Redis 预热与 Failover 容灾

# 系统初始化与预热阶段 ( EpochInitializer.java )

  • 触发时机：Spring Boot 微服务集群启动时。
  • 执行流程：
    1. 每一个服务实例启动时，都会去 Redis 检查 LUA_EPOCH 这个 Key 是否存在。
    2. 如果由于 Redis 刚重启或全面崩溃导致该 Key 不存在，Java 端会去 MySQL 查询当前最大的库存版本号（作为 maxMysqlEpoch ），并将其 +1 作为初始纪元 initEpoch 。
    3. 调用 batch_preload_stock.lua 脚本，将前 500 个高频核心商品的版本 Key 初始化为 newEpoch|0 ，per 流水号 seq 初始化为 0 。注意：系统使用了 EXISTS 校验，绝不覆盖运行期的真实活跃数据。

# 正常运行与心跳阶段 ( RedisHeartbeatRefresher.java )

  • 执行流程：
    • 微服务集群中的每个实例，每隔 5 秒 就会雷打不动地向 Redis 写入 / 刷新一个心跳 Key ( REDIS_HEARTBEAT_KEY )，并设置 TTL = 15秒 。
    • 只要 Redis 主节点正常存活，这个心跳 Key 就会永远存在（因为 5s < 15s，永远在到期前被续命）。

# 宕机与容灾升级阶段 ( RedisFailoverDetector.java )

  • 触发双探针：
    • 探针 A（被动事件）：Redisson 客户端监听到连接断开、重连或拓扑结构变化，触发监听器将 topologyDirtyFlag 设为 true 。
    • 探针 B（主动轮询）：看门狗线程每 5 秒扫描一次，一旦发现 !redisTemplate.hasKey(HEARTBEAT_KEY) （即心跳 Key 意外过期）或者 topologyDirtyFlag == true ，立刻判定系统进入 Failover 容灾自愈流程。
  • 原子升级 ( failover_epoch_upgrade.lua )：
    • 多个微服务实例的看门狗会同时冲向新选举出来的 Redis Master 节点执行升级脚本。
    • 第一个到达的实例：发现 doneKey 不存在，成功执行 INCR LUA_EPOCH （代数往前走一代），并立马写入一个 doneKey （设置 TTL = lockTTL * 2 = 90秒 ）。
    • 后续到达的实例：发现 doneKey 已经存在，说明同一次宕机事件已经被兄弟实例处理过了，直接返回 0 退出，成功拦截了群发风暴。

# 🧩 核心疑难问题深入拆解

# 问题一：90 秒盲窗期内连续二次宕机，消息会漏吗？

你的担忧：Master A 挂了 -> 触发 Epoch 升级并写了 doneKey （TTL 90s） -> 结果 10 秒内新主节点 Master B 又挂了。此时 doneKey 还在，是不是就无法升级 Epoch 了？

答案是：绝对不会漏，因为 Redis 的主从复制是异步的，这个物理现实反而帮了我们！

我们用精确的时序推演来看看第二次宕机发生时，新主节点（Master C）上到底发生了什么：

  1. 第一次切换：Master A 挂了，Slave B 被提升为 Master B。
  2. 微服务检测到变化，在 Master B 上成功执行了 Lua 脚本，把 LUA_EPOCH 顶到了 3 ，并且创建了 doneKey （TTL 90s）。
  3. 关键点（高并发漏同步）：由于 Redis 主从复制是异步的，Master B 刚刚写入 doneKey 和 LUA_EPOCH=3 ，还没来得及同步给它的从节点 Slave C，Master B 就突然物理烧毁（第二次宕机）！
  4. 第二次切换：Slave C 被提升为新的 Master C。
  5. 奇迹发生了：由于没来得及同步，现在的 Master C 身上根本没有 doneKey ！甚至连 LUA_EPOCH 都可能还是旧的值。
  6. 当微服务的看门狗再次杀到 Master C 时，执行 EXISTS doneKey 返回的是 0（不存在）！
  7. 结论：第二次升级会 100% 毫无阻碍地成功触发，你担心的 90 秒盲窗期在数据丢失的场景下会自动瓦解！

💡 补充一种极其罕见的极端情况：如果 Master B 活了 5 秒，把 doneKey 成功同步给了 Slave C 之后才挂的呢？ 如果 doneKey 成功同步到了 Slave C，说明 Master B 上的所有最新库存写操作、商品版本号，也已经 100% 完好无损地同步到了 Slave C。既然数据毫无丢失，那么在接下来的 85 秒内哪怕复用同一个 Epoch 继续运行，逻辑也是完全无懈可击的，根本不需要再次升级。

# 问题二：连接池抖动（假摔），会锁死真正的容灾吗？

你的担忧：如果没有发生宕机，只是单纯连接池重连，触发了 topologyDirtyFlag 。此时没有 doneKey 挡着，会造成一次无效的 Epoch+1。并且由于写了 doneKey 锁死 90 秒，导致在这 90 秒内如果发生真正的宕机，就无法有效升级了。 你的猜测：是不是说明 Epoch 只要发生过 +1 就可以，确保凌晨对账能发现就行，而不是去精准统计宕机几次？

分布式容灾设计的哲学本质

首先，Epoch 的目的绝对不是为了 “精准统计 Redis 到底挂了几次”。在架构设计中，Epoch 的本质是一个 “状态过期令牌（Generation Token）”。 只要代数往前走了（哪怕是因为网络假摔而误触发的 +1 ），旧的缓存版本（如 2|seq ）就会在逻辑上小于最新的全局纪元（如 3 ），这会逼迫接下来的业务请求通过 batch_lazy_load_stock.lua 去 MySQL 重新拉一次最新库存到 Redis。这在架构上叫 “保守防御”，代价只是让 MySQL 承受零星几次无伤大雅的查询，换来的却是缓存数据的绝对安全。

至于你担心的 “假摔后写了 doneKey ，90 秒内真挂了怎么办”：

  1. 网络假摔：集群其实完好，Master A 还在。看门狗误判，在 Master A 上把 Epoch 从 2 升到了 3，并种下了 doneKey 。
  2. 真挂发生：10 秒后，Master A 真的暴毙了，Slave B 顶上成为新 Master B。
  3. 结局依然完美：正如我们在问题一中分析的，Master A 猝死时，那个误创建的 doneKey 大概率还没同步给 Slave B。新主节点 Master B 盘子里是干净的（没有 doneKey ），看门狗冲过去依然能顺利把 Epoch 从 3 升级到 4！
# 那么 doneKey 的真正唯一作用是什么？

它是为了防御 “群发效应 / 回调风暴（Thundering Herd）”。 想象一下，你有 50 个 Spring Boot 实例。当网络突然抖动那一下，这 50 个实例会同时监听到连接变化。如果没有 doneKey 挡着，这 50 个实例会在同一秒内疯狂调用 Redis，把全局 Epoch 连刷 50 次（从 2 瞬间狂飙到 52）！ 这就会导致在线上运行的所有核心商品的版本号全部瞬间失效，引发几十万笔业务请求在同一毫秒全部穿透到 MySQL 去做懒加载重建，一瞬间就能把数据库当场拍死。 所以， doneKey 是个 90 秒的限流防撞墙，它确保 “同一波网络动荡，大家只能把代数往前推一代”，而不是用来精准计数。

# 问题三：主从切换太慢，会导致 Epoch 升级逻辑彻底失败并漏掉吗？

你的担忧：如果 Redis 节点物理损坏，主从切换和集群选举非常慢（比如花了 30 秒才选出新主）。在此期间，看门狗一直在每 5 秒运行一次，那这时候执行 Lua 脚本肯定全部爆红报错，会不会导致最后彻底漏掉 Epoch+1？

答案是：绝对不会漏掉，因为看门狗是 “状态驱动” 的，具备天然的持续重试机制。

我们来看一下当 Redis 正在艰难选举（处于无主状态）时，系统的运作状态：

  1. 选举盲窗期（第 0 ~ 25 秒）：

    • Redis 集群处于 CLUSTERDOWN 或只读状态，无法接受写命令。

    • 微服务实例的看门狗每 5 秒醒来一次，高喊着要执行 failover_epoch_upgrade.lua 。

    • 结果：Redis 拒绝写入，Lua 脚本抛出异常（连接失败或集群不可用）。

    • RedisFailoverDetector.java 内部抓住了这个异常：

    • 关键点：由于执行失败，Java 线程绝对不会去清除 topologyDirtyFlag （它依然是 true ），而且由于 Redis 无法写入， HEARTBEAT_KEY 也绝对无法被刷新（它依然是不存在的）。

  2. 拨云见日，新主登基（第 30 秒）：

    • Redis 集群选举成功，新 Master 节点开始正常接收写请求。

  3. 完美收网（第 35 秒）：

    • 下一个 5 秒周期到了，看门狗再次醒来。
    • 检查条件： topologyDirtyFlag 依旧是 true ！心跳 Key 依旧不存在！
    • 看门狗再次发起 Lua 脚本请求。这一次，新 Master 满血复活，欢快地接受了请求，成功将 Epoch + 1 并在新主上写下了 doneKey 。
    • 升级成功，代码执行 heartbeatRefresher.refresh() 恢复正常心跳，清空脏标志位，自愈完成。

总结：只要 Redis 一天不恢复响应，看门狗就会死磕到底（每 5 秒冲锋一次）。在 Redis 瘫痪的盲窗期内，业务侧由于拿不到 Redis 连接，扣库存请求本来就无法进行，因而也不会产生任何脏数据。系统升级 Epoch 的动作可能会 “迟到”（等待集群选举完毕），但由于状态标志位的保护，它绝对不会 “缺席”。
